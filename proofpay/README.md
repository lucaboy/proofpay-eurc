# ProofPay EURC

ProofPay is a ZeroClaw 0.8.3 skill-only workflow for freelancers and studios:
bind an exact deliverable revision to an EURC Solana Pay request, let the client
sign in their own wallet, then reconcile the finalized transfer into a
reproducible evidence pack.

The agent has no wallet. It never holds a seed phrase or private key, never
signs or submits a transaction, and has no send/refund command.

## Why this exists

Payment links answer “where and how much?” but usually lose the connection to
the exact work revision being paid. ProofPay adds a verifiable, privacy-aware
commitment:

```text
invoice + recipient + amount + EURC mint + deliverable SHA-256
                              |
                              v
                  human-approved Solana Pay URI
                              |
                 payer signs in their own wallet
                              |
                              v
 finalized transaction + exact checks + local evidence pack
```

A buyer can see normal Solana Pay fields. The operator retains the full
SHA-256 and request record. The resulting evidence says only that one finalized
EURC transfer matched one immutable technical request; it does not pretend to
be legal acceptance or identity proof.

## Safety by construction

- **Zero custody:** no key storage, wallet adapter, signing session, transfer,
  swap, bridge, or refund path.
- **Human gate:** recipient, amount, network, fixed mint, file, and digest are
  displayed before an out-of-band operator checkpoint. `create` requires the
  exact digest, reference, and URI returned by the approved preview and records
  their technical preview-match commitment.
- **Fixed asset:** the Circle-listed Solana EURC mint is pinned to
  `HzwqbKZw8HxMN6bF2yFZNrht3c2iXXzpKcFu7uBEDKtr`; token symbols are never
  trusted.
- **Exact arithmetic:** EURC decimal amounts are converted to atomic units
  without floating point.
- **Opaque deliverables:** files are hashed as bytes and never executed or
  interpreted as prompts.
- **Prompt-injection resistance:** customer text, file content, memos, and RPC
  fields are untrusted data and cannot change authority or payment terms.
- **Fail-closed verification:** finality, success, the unique successful
  finalized reference match, network, mint, recipient owner, canonical
  associated token destination, exact amount, compiled transfer binding, and
  local signature uniqueness must all match. A non-null transaction
  `blockTime` must also fall inside the fixed payment window (with bounded clock
  skew) and agree across the signature-history and transaction responses.
- **Serialized persistence:** a filesystem lock permits one ledger writer at a
  time; creation and paid-state transitions revalidate immutable terms inside
  that lock.
- **Application-level no-overwrite evidence:** an invoice evidence bundle is
  created exclusively and is never silently replaced or regenerated by the
  helper. An operating-system owner can still edit or delete local files.
- **Bounded attribution:** evidence schema v3 includes the persisted
  preview-match commitment, `validForSeconds`, `expiresAt`, and
  `withinPaymentWindow`, but does not claim who approved it; human checkpoint
  identity remains in the external operator/SOP audit trail.
- **Private public metadata:** only a non-sensitive invoice slug and a short
  hash commitment enter the memo—never names, emails, paths, or contract text.
- **Skill-only package:** `manifest.toml` declares only `skill`, no WASM, and
  zero plugin permissions.
- **Runtime dispatch evidence:** the sanitized ZeroClaw trace must contain both
  a parsed native tool call and the actual returned fixed-helper JSON. This
  rejects prose-only claims, while making no receipt, signature, or human
  attribution claim for the stock v0.8.3 direct CLI path.

Read the complete [threat model](./docs/THREAT_MODEL.md) before mainnet use.
The [standards crosswalk](./docs/STANDARDS.md) maps each interoperability and
verification claim to the primary ZeroClaw, Solana Pay, Solana RPC, or Circle
source and to the corresponding ProofPay control.
The [research matrix](./docs/RESEARCH.md) records the agent-payment papers,
security guidance, open-source comparisons, adopted controls, and explicit
non-claims.

## Architecture

```text
                         CONTROL PLANE

 Operator ── terms ──> ZeroClaw `proofpay` agent
    ^                   │
    │ out-of-band       ├── proofpay-eurc SKILL.md
    │ approval          ├── create-request SOP (checkpoint)
    └───────────────────└── reconcile SOP (manual / cron)
                               │ allowlisted helper invocation
                               v
                      ProofPay local helper
                         │            │
                  request ledger   evidence pack

                          PAYMENT PLANE

 Solana Pay URI ──> payer wallet ── signs independently ──> Solana
                                                            │
                              allowlisted read-only RPC <────┘

 There is intentionally no path from the agent/helper to a signing key.
```

The integration is Tier 1: stock ZeroClaw skill instructions, supervised SOP
references, CLI channel, and one dependency-light local helper. No custom
ZeroClaw fork, WASM plugin, MCP server, or wallet SDK is required. The locked
demo profile exposes no raw `shell`: its six manifest-locked wrappers are three
non-mutating views (`hash_sample`, `preview_sample`, and
`list_local_requests`), one always-ask idempotent request writer
(`create_sample_request`), one fixed reconciler that may persist a verified
`pending` to `paid` transition (`check_sample_payment`), and one exclusive
evidence writer (`write_sample_evidence`). None can sign, submit, transfer, or
refund funds. The model can also inspect the five built-in SOP control verbs,
but general dynamic SOP shell steps remain documentation for an
operator-managed deployment and are not executable through this locked demo
profile.

The demo publishes a sanitized trace summary only after it finds both a parsed
native tool call and the actual fixed-helper result in the same ZeroClaw trace.
This is reproducible evidence against model narration—not a cryptographic
receipt, an external attestation, or a substitute for the visible approval
gate.

## What gets bound

Each request stores:

- a non-sensitive invoice slug, normalized to lowercase before preview,
  reference derivation, memo construction, and persistence;
- `devnet` or `mainnet`;
- the operator’s full recipient public key;
- an exact EURC amount;
- the fixed EURC mint;
- a fixed validity duration of 604800 seconds (seven days);
- SHA-256 of an exact file inside `proofpay/deliverables/`;
- a deterministic, domain-separated Solana reference derived from the complete
  approved terms, full deliverable digest, and fixed validity duration;
- a deterministic public memo:
  `PROOFPAY:<invoice>:<first-16-sha256>`;
- the generated Solana Pay URI and reconciliation state;
- a persisted `preview-match` commitment containing the exact digest, reference,
  and URI received by `create`, plus its local recording time;
- an absolute `expiresAt` computed when `create` records the request.

The reference binds the fixed duration, not the preview’s wall-clock issue time:
Solana Pay transfer URLs have no native expiry field. Preview freshness therefore
relies on the ZeroClaw `always_ask` checkpoint at the actual
`create_sample_request` invocation. A payer can still submit a late on-chain
transfer, but ProofPay will not accept one whose `blockTime` is after
`expiresAt` plus the five-minute skew allowance.

Changing the deliverable creates a new digest. Do not rewrite an already paid
revision; issue a new invoice/revision instead.

## Quick start

Requirements: Node.js 16.10+ and the ZeroClaw 0.8.3 binary.

Run all offline tests:

```sh
npm test
```

Prepare an isolated config, then audit the two skills copied into its declared
bundle:

```sh
sh ./proofpay/demo/prepare-zeroclaw-demo.sh proofpay-demo

zeroclaw skills audit \
  --config-dir .runtime/proofpay-demo \
  .runtime/proofpay-demo/shared/skills/proofpay/proofpay-eurc

zeroclaw skills audit \
  --config-dir .runtime/proofpay-demo \
  .runtime/proofpay-demo/shared/skills/proofpay/proofpay-demo-tools

zeroclaw skills list \
  --config-dir .runtime/proofpay-demo \
  --agent proofpay
```

The preparation script creates a dedicated config-owned workspace and copies
only the reviewed helper, sample, SOPs, and both skills (`proofpay-eurc` and
`proofpay-demo`). It refuses to overwrite an existing instance. Validate its
copied SOPs:

```sh
zeroclaw sop validate \
  --config-dir .runtime/proofpay-demo \
  proofpay-create-request

zeroclaw sop validate \
  --config-dir .runtime/proofpay-demo \
  proofpay-reconcile

zeroclaw sop graph \
  --config-dir .runtime/proofpay-demo \
  proofpay-create-request
```

Run the preparation command from the repository root. Run ZeroClaw commands
against the isolated config directory shown above; its helper operates in the
copied, config-owned workspace rather than the checkout. The template uses an
OpenAI Codex subscription slot without embedding credentials; login remains an
explicit operator action.

Full clean-room instructions are in
[`docs/REPRODUCE.md`](./docs/REPRODUCE.md).
Tested-source bundle reproducibility and GitHub attestation scope are in
[`docs/SUPPLY_CHAIN.md`](./docs/SUPPLY_CHAIN.md).

On macOS, keep the template’s `sandbox_backend = "auto"` first. Stock ZeroClaw
0.8.3 Seatbelt can reject Node while traversing ancestor paths, even in an
isolated workspace. For a trusted local recording only, the preparation script
supports a separately generated `/private/tmp` runtime with an explicit
`--local-no-os-sandbox` warning; do not deploy or share that fallback runtime.

## Request-creation demo

Set only a public Solana address you control:

```sh
export PROOFPAY_RECIPIENT="<SOLANA_PUBLIC_ADDRESS>"
bash ./proofpay/demo/run-demo.sh
```

This local script demonstrates the approval and request-creation phase only. It
hashes
[`deliverables/sample-milestone.txt`](./deliverables/sample-milestone.txt),
captures the devnet preview in a private temporary file, displays it, and waits
for typed approval. Without `jq`, it parses the preview with Node.js and passes
the exact returned digest, reference, and URI to `create`. The helper re-hashes
and re-derives the request; any difference fails before persistence. The script
does not open a wallet, sign, submit, send, or refund.

The committed sample’s expected SHA-256 is:

```text
4a3adafc3eeaa1670c5acd78349af5db9755c89efa0f9015f9bc293392ec20c8
```

Preview the same request without persistence:

```sh
./proofpay/tools/proofpay.mjs preview \
  --invoice demo-atlas-m1 \
  --recipient "${PROOFPAY_RECIPIENT}" \
  --amount 12.50 \
  --network devnet \
  --deliverable sample-milestone.txt
```

Create only after comparing the preview and approving the exact terms:

```sh
# Copy these three values verbatim from preview.approval.
APPROVED_DIGEST="<preview.approval.deliverableSha256>"
APPROVED_REFERENCE="<preview.approval.reference>"
APPROVED_URI="<preview.approval.solanaPayUri>"

./proofpay/tools/proofpay.mjs create \
  --invoice demo-atlas-m1 \
  --recipient "${PROOFPAY_RECIPIENT}" \
  --amount 12.50 \
  --network devnet \
  --deliverable sample-milestone.txt \
  --approve-digest "${APPROVED_DIGEST}" \
  --approve-reference "${APPROVED_REFERENCE}" \
  --approve-uri "${APPROVED_URI}"
```

The payer reviews and signs the URI in an external wallet. That step is outside
ProofPay. The final submission demo additionally shows a real devnet payment
followed by the fixed ZeroClaw reconciliation and evidence tools.

## CLI surface

The helper intentionally exposes a small command set:

```text
preview  validate terms, hash bytes, and emit approval values without persistence
hash     hash one contained deliverable as opaque bytes
create   require exact preview approval values, then idempotently persist one request
list     list local requests; add --compact for bounded agent output
check    reconcile one invoice through read-only Solana RPC
evidence write/display evidence for an invoice that reconciled as paid
verify-evidence independently validate an evidence file and deliverable offline,
                or add --online to re-query Solana
```

Examples:

```sh
./proofpay/tools/proofpay.mjs hash \
  --deliverable sample-milestone.txt

./proofpay/tools/proofpay.mjs list --compact --json

./proofpay/tools/proofpay.mjs check \
  --invoice demo-atlas-m1 \
  --compact --json

./proofpay/tools/proofpay.mjs evidence \
  --invoice demo-atlas-m1 \
  --compact

./proofpay/tools/proofpay.mjs verify-evidence \
  --evidence proofpay/evidence/demo-atlas-m1.evidence/evidence.json \
  --deliverable proofpay/deliverables/sample-milestone.txt

# Re-run the same canonical checks against the allowlisted Solana RPC.
./proofpay/tools/proofpay.mjs verify-evidence \
  --evidence proofpay/evidence/demo-atlas-m1.evidence/evidence.json \
  --deliverable proofpay/deliverables/sample-milestone.txt \
  --online
```

Offline verification checks the exact schema, canonical request derivation,
timestamps, required limitations, and supplied deliverable bytes. It does not
query Solana or authenticate the evidence producer. `--online` additionally
re-runs the payment lookup and exact transaction checks against the request’s
allowlisted network RPC; it still does not authenticate the producer.

There is deliberately no `sign`, `send`, `pay`, `transfer`, `swap`, `refund`,
`import-key`, or arbitrary RPC command.

## ZeroClaw workflows

### Create request

The `proofpay-create-request` SOP:

1. validates inputs and runs a non-persistent preview;
2. shows full normalized terms, the fixed validity duration, and the three
   exact approval values;
3. pauses at an out-of-band operator checkpoint;
4. passes the approved digest, reference, and URI verbatim to `create`;
5. re-hashes/re-derives and creates the immutable local request only if all
   three still match, persisting a technical preview-match commitment and
   absolute `expiresAt`;
6. hands off the URI while clearly stating that no transaction was submitted.

An exact retry returns the existing request with `idempotent: true` and does not
rewrite timestamps or state. Reusing the same canonical invoice ID with any
different immutable term fails with `INVOICE_CONFLICT`. Any changed field
requires a new invoice/revision, preview, and approval.

### Reconcile

The `proofpay-reconcile` SOP can run manually or on its 15-minute cron trigger.
It:

1. loads at most 20 pending, unexpired local invoice slugs from compact output;
2. checks them individually through the narrow helper;
3. preserves stored immutable state; typed mismatches/errors do not rewrite it,
   while an overdue pending record is displayed as the derived status
   `expired`;
4. reports locally and never publishes or messages evidence.

Cron activity requires the ZeroClaw daemon/SOP scheduler; installing the files
alone does not start a background process.

## What “paid” means

ProofPay reports `paid` only when one finalized, successful Solana transaction
matches every stored property:

| Check | Required fact |
|---|---|
| Network | RPC and request use the same pinned network |
| Finality | Transaction is `finalized` |
| Execution | Transaction metadata reports no error |
| Reference | Exactly one successful finalized signature is returned for the stored reference; that reference is a readonly non-signer transaction key and the additional account on the compiled final SPL transfer |
| Mint | Exact pinned EURC mint, not a display symbol |
| Recipient | Credited token-account owner equals stored recipient and the transfer destination is the canonical associated token account derived from recipient plus mint |
| Amount | Exact EURC atomic-unit delta |
| Memo | Exactly one stored invoice/hash commitment matches |
| Instruction tail | Exact memo is penultimate; compiled SPL `transfer`/`transferChecked` is final, carries the reference, and targets the canonical associated token account |
| Replay | Signature has not settled another local request |
| Time window | Non-null `blockTime` agrees across RPC views and falls from five minutes before request creation through the earlier of five minutes after `expiresAt` and five minutes after verification |

A transaction signature by itself is not enough. See
[`docs/EVIDENCE.md`](./docs/EVIDENCE.md) for the evidence schema and independent
verification procedure.

Evidence schema v3 carries the stored preview-match commitment,
`validForSeconds`, `expiresAt`, `uniqueSuccessfulFinalizedReference`,
`withinPaymentWindow`, and `previewMatchedAtCreation`. This establishes
technical equality and bounded payment timing; it is not an identity claim or a
human checkpoint signature.

committed with sensitive business data. Ledger changes are serialized by a

rerunning `evidence` cannot overwrite an existing bundle.

## Limitations

- A compromised operator device or payer wallet remains dangerous.
- One RPC can lie or fail; production users should consider a second
  independent RPC or verified node.
- The short on-chain hash prefix is a correlation aid; the full local SHA-256
  is the actual deliverable commitment.
- Solana Pay cannot prevent a payer broadcasting after `expiresAt`; ProofPay
  rejects a late match but cannot reverse that on-chain transfer.
- ProofPay does not prove identity, authorship, work quality, legal acceptance,
  sanctions compliance, tax treatment, or refund entitlement.
- EURC retains issuer, freeze, depeg, smart-contract, legal, and availability
  risk.
- Public Solana metadata is permanent; use a non-sensitive invoice slug.
